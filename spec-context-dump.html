Estado completo do sistema em Junho 2026. 25+ módulos. 4 camadas. 1 homepage.

● 1

Arquitetura Atual

🌐 API Layer — api-football.ts

Cliente HTTP para API-Football v3. Fetch de fixtures (por data, ao vivo), standings, ligas. Fallback silencioso se API_FOOTBALL_KEY não estiver configurada.

Inputs: env vars • Outputs: ApiFixture[] • Deps: fetch

⚙️ Transformers — transformers.ts

Converte ApiFixture → Match. Mapeia status, estágio, penalidades. Calcula importanceScore (pesos: fase, popularidade, gols, virada).

Inputs: ApiFixture • Outputs: Match • Deps: api-football.ts

📊 Context Builder — context-builder.ts

Enriquece Match → EnrichedMatch. Adiciona narrativeFlags (upset, penalty_drama, blowout), storyCandidates, competitionImpact, simpleImpact, historicalContext, teamForm.

Inputs: Match[] • Outputs: EnrichedMatch[] • Deps: types

📜 Historical Facts — historical-context-builder.ts

Detecta fatos históricos a partir de EnrichedMatch: first_african_semifinalist, giant_killing, years_since_last_title, defending_champion_eliminated, streak_broken. Cada fato tem editorialWeight.

Inputs: EnrichedMatch • Outputs: HistoricalFact[] • Deps: knowledge.ts

🏛️ Knowledge Base — knowledge.ts

Dataset hardcoded com títulos por seleção, FIFA ranking, melhor resultado histórico, última conquista. Usado por historical-context-builder para detecção de fatos.

Inputs: team name • Outputs: {titles, rank, bestResult, lastTitle} • Deps: nenhuma

🧠 Tournament Memory — tournament-memory.ts

Constrói a memória do torneio: teamJourneys (texto descritivo por time), narrativas detectadas (redemption, champion recovery), teamForm, historicalContexts, narrativeArcs.

Inputs: Match[] • Outputs: TournamentMemory • Deps: narrative-accumulation

📈 Narrative Accumulation — narrative-accumulation.ts

Detecta padrões narrativos em múltiplos jogos: multi_upset_run (2+ vitórias sobre favoritos), cinderella_progression (vitórias consecutivas em mata-mata), redemption_journey (derrota na estreia → campeão), giant_slayer (2+ favoritos eliminados).

Inputs: Match[] • Outputs: NarrativeArc[] • Deps: types

🎯 Editorial Story Engine — editorial-story-engine.ts

Módulo central de seleção. Para cada partida: classifica storyType, calcula confidence (0-1), gera evidence[], headline, narrativeHook, whyItMatters. Ordena por confidence. Define priority 1 (hero), 2 (secondary), 3 (emerging).

Inputs: EnrichedMatch[], TournamentMemory • Outputs: EditorialStory[] • Deps: tudo

📋 StoryBrief — story-brief.ts

Gera o resumo de 15 segundos: headline + 3 bullets + continuity (dia, fase, ontem, hoje). Entidade central de consumo do produto.

Inputs: EditorialStory[], TournamentMemory • Outputs: StoryBrief

📖 NextChapter — next-chapter.ts

Gera o gancho de retenção: headline + hook + openQuestion + próximo evento. Cada tipo de arco tem uma pergunta aberta específica. Só gera se hasOpenQuestion === true.

Inputs: StoryBrief, NarrativeArc[] • Outputs: NextChapter | null

📊 Narrative Tracker — narrative-tracker.ts

Transforma NarrativeArc[] em ActiveNarrative[] visível ao usuário. Calcula capítulo atual, status (active/completed/archived), próximo passo. Máximo 3 narrativas.

Inputs: NarrativeArc[], TournamentMemory • Outputs: ActiveNarrative[]

● 4

Sistema Editorial

Como uma história é escolhida

selectStories() em editorial-story-engine.ts percorre todos os EnrichedMatch finalizados e para cada um:

  1. Classifica storyType — por ordem de prioridade: editorial event → narrative arc → historical fact → match flag
  2. Calcula confidence (0-1): base 0.1 + eventos editoriais (+0.8-0.9) + arcos narrativos (+0.6-0.9) + fatos históricos (+peso/500) + estágio (+0.2-0.5) + popular eliminado (+0.3) + pênaltis (+0.15) + zebra (+0.15). Cap em 1.0.
  3. Gera evidence[] — lista legível de razões. "Brasil — uma das maiores seleções — foi eliminado." "É a final da Copa." "Decisão dramática nos pênaltis."
  4. Ordena por confidence decrescente
  5. Atribui priority: 1 (hero), 2 (secondary), 3 (emerging)

Hero Story (priority 1)

Maior confidence. Se há final, é sempre hero. Se há editorialEvent com weight ≥ 90, garante top 3. Se há arco narrativo, ganha bônus.

Secondary (priority 2)

Segundo maior confidence. Pode ser outro jogo importante do dia. Se só há 1 jogo, pode não existir.

Emerging (priority 3)

Terceiro maior confidence. Geralmente um jogo ao vivo ou o próximo jogo agendado de maior importância.

● 7

Tournament Memory

🔧 Como é construída

buildMemory(matches) em tournament-memory.ts: recebe todos os Match do torneio, agrupa por time, ordena por data. Para cada time, gera descrições textuais de cada partida ("Eliminou Portugal por 1-0").

🔄 Como é atualizada

A cada fetch da API (60s), os matches são refetchados e a memória é reconstruída do zero. Não há persistência entre builds. É puramente funcional.

📦 O que armazena

  • teamJourneys: Record — texto descritivo por time. Ex: "Argentina: Perdeu para Arábia → Eliminou Holanda nos pênaltis → Eliminou Croácia → Eliminou França nos pênaltis"
  • narratives: string[] — frases detectadas. Ex: "Argentina perdeu na estreia mas se recuperou para ser campeão"
  • teamForm: Record — lost_opener, current_streak, momentum
  • historicalContexts: Record — first_african_semifinalist, etc
  • historicalFacts: HistoricalFact[] — fatos detectados (shared com prompt)
  • narrativeArcs: NarrativeArc[] — arcos detectados

🔗 Como alimenta o sistema

  • teamJourneys → prompt da LLM (contexto narrativo)
  • narratives → prompt da LLM (contexto narrativo)
  • narrativeArcs → editorial-story-engine (confidence boost) + next-chapter (geração) + narrative-tracker (exibição)
  • historicalFacts → editorial-story-engine (confidence boost) + prompt LLM
● 8

Homepage Atual

1. 🏁 ContinuityBar

Topo da página. Mostra dia da Copa, fase atual, número de jogos hoje, ontem (time vencedor) e hoje (próximo jogo). Gera sensação de progressão.

2. 📋 QuickRead

Headline editorial + 3 bullets. 15 segundos de leitura. A unidade principal de consumo. "Marrocos faz história como primeira seleção africana na semifinal" + 3 bullets com resumo de cada jogo.

3. 📖 NextChapter

Condicional (só aparece se hasOpenQuestion). Mostra o próximo capítulo da história ativa. "Marrocos tenta continuar campanha. Será que a zebra chega à final?" — mecanismo de retenção.

4. 📊 NarrativeTracker

Lista as histórias ativas da Copa (máx 3). Cada uma mostra capítulo atual, status, próximo passo. "Marrocos: Campanha histórica — Cap 4/6 — Próximo: França".

5. 🔥 Hero Story

Card editorial com headline, tag (🔥 Redenção, 🌍 Marco), why it matters, confidence. É a história mais importante do dia selecionada pelo editorial-story-engine.

6. ⚽ Matches

Jogos ao vivo (com indicador pulsante) + próximos. Layout horizontal, 2 cards por linha. Mobile abaixo do Hero. Desktop na sidebar.

7. 📊 Standings

Tabela compacta do grupo mais relevante. Máximo 4 linhas. Seletor de grupos em pills. Abaixo dos jogos na sidebar desktop. Abaixo de tudo no mobile.

📱 Layout

Mobile: 1 coluna, ordem vertical. Desktop: 2 colunas (1.5fr / 1fr), conteúdos na esquerda, sidebar (Matches + NarrativeTracker + Standings + Footer) na direita.

● 9

Débitos Técnicos

🐛 Problemas conhecidos

  • mapStage() em transformers.ts e validate.mjs — ordem das checagens pode mapear "Quarter-finals" como "Final" (contém "final"). Corrigido mas duplicado.
  • Node.js 18.14.0 é antigo — Next.js 16 requer ≥20.9. Preso no Next.js 13. npx next build do PATH global tenta rodar Next 16.
  • Cache do .next causa EPERM no Windows com frequência — precisa de Remove-Item .next -Recurse -Force antes de rebuild.
  • Scripts standalone (.mjs) não leem .env.local — precisam de variáveis passadas no terminal.

🗑️ Campos não utilizados

  • simpleImpact.* (classification, consequence, tournamentEffect) — nunca enviado à LLM, apenas usado internamente por templates
  • competitionImpact.nextOpponent — sempre null, nunca implementado
  • storyCandidates[].score — enviado ao prompt mas LLM ignora números. Só o type importa
  • Bulletin — mantido por compatibilidade, substituído por EditorialStory + StoryBrief
  • heroImportanceScore — duplicado pelo confidence do EditorialStory

🔧 Melhorias identificadas

  • Player Facts (last_dance, legacy_complete, injury, hero_performance) — especificados mas não implementados
  • hat_trick detection — pode ser adicionado ao player-facts sem refatoração
  • Contexto entre jogos do mesmo time — detectado (narrativeArcs), mas não visível no NarrativeTracker como "capítulo anterior"
  • Áudio — especificado (spec-script-generator.html) mas não integrado à homepage
  • StoryBrief.shareText — implementado mas sem integração com share button funcional
  • OG images — especificadas em spec-story-package.html mas não geradas (@vercel/og pendente)
● 10

Próxima Fronteira: Contexto Histórico

❌ Bases históricas ausentes

  • Histórico de confrontos entre seleções: Quem eliminou quem em Copas anteriores. Sem isso, não detectamos revenge_match, repeat_final, rematch.
  • Recordes individuais históricos: Maiores artilheiros, mais jogos, mais assistências. Sem dados de jogadores.
  • Linha do tempo de todas as Copas: Campeões, vices, artilheiros, países-sede, fato marcante. O knowledge.ts só tem títulos.
  • Ranking histórico (ELO ou similar): Saber se uma zebra de 2022 é maior ou menor que uma zebra de 1950.
  • Dados demográficos de jogadores: Idade, altura, posição, clube. Essencial para detectar jogador mais jovem, mais velho, revelação.

📂 Knowledge files que deveriam existir

  • world-cup-history.json — todas as Copas com campeão, vice, artilheiro, país-sede, ano. ~100 linhas.
  • team-rivalries.json — pares de seleções com histórico de confronto em Copas. ~20 linhas.
  • player-legends.json — Messi, CR7, Neymar, Mbappé, Modric, etc. ~20 linhas (parcialmente especificado em player-roster).
  • milestone-database.json — marcos específicos como "primeira final entre sul-americanos", "primeira final repetida", etc. ~30 linhas.

📜 Fatos históricos não detectáveis hoje

  • repeat_final — mesma final de Copa anterior. Precisa de world-cup-history.json.
  • revenge_match — time A eliminou time B em Copa anterior. Precisa de histórico de confrontos.
  • first_title_ever — time conquista a primeira Copa. Precisa de contar títulos (knowledge.ts já tem, mas detector não existe).
  • back_to_back_final — mesma seleção em duas finais consecutivas. Precisa de world-cup-history.json.
  • host_nation_success / host_nation_eliminated — precisa de flag "host" no knowledge.ts.
  • threepeat_attempt — time busca o tri consecutivo. Precisa de world-cup-history.json com últimos 2 campeões.

📖 Narrativas históricas não geráveis hoje

  • "Maior zebra da história": Não podemos comparar zebras entre edições porque não temos ranking histórico.
  • "Primeira vez que X acontece desde 1930": Sem a linha do tempo completa, não sabemos o que é inédito.
  • "Jogador X se iguala a Pelé/Y": Sem dados individuais históricos, não podemos fazer comparações.
  • "Esta é a Copa com mais gols desde 1958": Sem dados agregados por edição, não podemos gerar contexto estatístico.
  • "A última vez que isso aconteceu foi em...": Qualquer comparação temporal exige world-cup-history.json.

🎯 Prioridade: world-cup-history.json

Um arquivo de ~100 linhas com todas as 22 edições da Copa (1930-2022) — ano, país-sede, campeão, vice, artilheiro, número de times, fato marcante. Com isso, o sistema pode detectar repeat_final, back_to_back_final, threepeat_attempt, revenge_match, e responder "primeira vez desde X". É o dataset de maior ROI para a próxima sprint de contexto histórico.
