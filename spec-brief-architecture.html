● 2

Entidade StoryBrief

interface StoryBrief {
  // Identificação
  id: string
  date: string

  // Quick summary — 10-14 palavras. O que importa HOJE.
  headline: string

  // 3 bullets. Cada um: 8-15 palavras.
  // Bullet 1: Hero Story com why it matters
  // Bullet 2: Secondary Story (ou jogo ao vivo)
  // Bullet 3: Emerging / Amanhã / O que esperar
  bullets: [string, string, string]

  // Contexto do torneio — gerado automaticamente
  continuity: {
    day: number           // Dia 23 da Copa
    phase: string         // Fase de grupos / Mata-mata
    phaseProgress: string // "3ª rodada" ou "Quartas"
    matchCount: number    // jogos hoje
    yesterday: string     // "Marrocos eliminou Portugal"
    today: string         // "Argentina x Croácia"
  }

  // Metadados editoriais
  storyType: EditorialStoryType
  tag: string
  confidence: number
  readingTime: number     // ≤ 20s sempre

  // Derivados — podem ser nulos se não existirem
  shareText: string | null
  audioAvailable: boolean
}
Por que StoryBrief é uma entidade (não uma view): Porque ela define o contrato de consumo do produto inteiro. A homepage, o push, a newsletter, o audio — todos partem do Brief. Se o Brief está errado, o produto inteiro está errado. É a primeira coisa que o sistema produz depois do EditorialStoryEngine. É a última coisa que o usuário consome antes de decidir se abre ou não o site.

Quick Read

📋 Especificação

interface QuickRead {
  // 1 headline + 3 bullets. Sempre.
  headline: string
  bullets: [string, string, string]

  // Barra de continuidade
  continuityLine: string  // "Dia 23 • Quartas • 2 jogos hoje"

  // Link para mais
  detailAvailable: boolean
  audioAvailable: boolean
}

Regra dos 3 bullets:

  • Bullet 1: Hero Story — headline + why it matters + time vencedor
  • Bullet 2: Secondary Story OU jogo ao vivo mais importante. Se não houver, contexto extra do Hero.
  • Bullet 3: Amanhã OU jogo ao vivo OU emerging. Sempre o futuro próximo.

● 6

StoryPackage: Manter ou Remover?

❌ Opção B: Remover

StoryBrief substitui StoryPackage. O sistema gera apenas o Brief. Quem quiser distribuir para newsletter/social/push faz isso a partir do Brief, não de um Package dedicado.

✅ Opção C: StoryPackage vira agregador

StoryBrief é a entidade principal. StoryPackage é uma view que agrega Brief + formatos derivados (share text, audio flag, etc.). Package existe como estrutura de dados, não como camada editorial.

Relação: StoryBrief ⊂ StoryPackage — Brief é obrigatório, Package é opcional.

Decisão: Opção C. StoryPackage vira agregador. O sistema produz um StoryBrief obrigatoriamente. Se houver necessidade de formatos adicionais, o Package é composto a partir do Brief. Zero lógica editorial nova. Apenas formatação de saída.

● 7

Density Score

📊 Como medir valor entregue por segundo

Density = evidence_count / reading_time

evidence_count: Número de fatos no StoryBrief (headline + 3 bullets = 4 unidades de informação)
reading_time: Tempo estimado de leitura em segundos

Exemplo Quick Read: 4 evidências / 15 segundos = 0.27 fatos/segundo
Exemplo Audio: 5 evidências / 90 segundos = 0.06 fatos/segundo
Exemplo Newsletter: 6 evidências / 120 segundos = 0.05 fatos/segundo

Meta: Density ≥ 0.2 (Quick Read). Tudo abaixo disso é formato de nicho.

● 8

Comparação de Formatos

📋 EditorialStory

Consumo: 0s (ninguém consome isso diretamente)

headline + confidence + evidence + narrativeHook + whyItMatters + storyType + tag + priority + matchId

📋 StoryBrief

Consumo: 15s — 0.27 fatos/s ✅

headline + 3 bullets + continuity + tag + shareText

📋 Quick Read

Consumo: 15s — 0.27 fatos/s ✅

É a VIEW do StoryBrief na homepage. Mesmo conteúdo, formato visual.

📋 Homepage (completa)

Consumo: 30s — 0.13 fatos/s 🟡

Brief + Hero + Matches + Standings. O Brief é a parte mais densa.

StoryBrief ganha. Maior density score. Menor tempo de consumo. Mesma qualidade editorial. É o formato que o usuário realmente consome.
