● 3

Entidade Player Narrative

PlayerFact
{
  id: "messi_hat_trick"
  playerId: "messi"
  playerName: "Lionel Messi"
  type: "legacy_complete" | "last_dance" | "hat_trick" | "injury" | "captain_title" | "goal_record" | "golden_boot" | "breakthrough"
  team: "Argentina"
  confidence: number // 0-1
  editorialWeight: number // 0-100
  description: string // texto legível
  triggerMatchId: string | null
}

A entidade é quase idêntica a HistoricalFact. A diferença: PlayerFact tem playerName e playerId, e pode ser vinculada a um jogo específico. Ela entra no EditorialStoryEngine exatamente como um HistoricalFact — como evidência que aumenta o confidence do story.

● 4

Fontes de Dados

✅ Dados estruturados (API-Football)

  • Escalação: Saber se um jogador está em campo → essencial para "última Copa"
  • Eventos: Gols, assistências, cartões, substituições → detectar hat-tricks, herói improvável, goleiro pegando pênalti
  • Lesões: API-Football tem endpoint de lesões → detectar Neymar 2022
  • Estatísticas: Artilharia, assistências, clean sheets → detectar artilheiro do torneio

Usando apenas dados que a API-Football já retorna. Zero custo adicional.

⏰ Dataset curado (player-roster.json)

  • 20-30 jogadores lendários ativos — Messi, CR7, Neymar, Mbappé, Modric, Lewandowski, De Bruyne, etc.
  • Fatos fixos por jogador: idade, última Copa, títulos que faltam, recordes que possui
  • Narrativa principal: "last_dance", "legacy_complete", "breakthrough"

Dataset manual de ~100 linhas. Atualizado a cada Copa.

❌ Não usar agora

  • NLP de notícias: Para declarações, polêmicas, conflitos. Exige redação, não algoritmo.
  • Redes sociais: Volume alto demais, ruído excessivo para um MVP.
  • Sportmonks / TheSportsDB: APIs adicionais com sobreposição à API-Football.
● 5

Team Stories + Player Stories

Regra de combinação: Player Stories NUNCA substituem Team Stories. Elas apenas adicionam evidência. Uma Team Story sem Player Story funciona. Uma Player Story sem Team Story não existe.
Team Story
Argentina campeã
+
Player Fact
Messi última Copa
→
EditorialStory
Argentina campeã + Messi legado

Exemplo concreto: A Team Story "Argentina campeã" já existe com confidence 0.97. Ao adicionar playerFact: "Messi conquista o único título que faltava" (editorialWeight: 98), o confidence sobe para 0.99 e o evidence[] ganha "Messi, o maior jogador da história, finalmente tem a Copa que merecia." — mas a headline continua sendo "Argentina campeã", não "Messi campeão". O time ainda é a história principal. O jogador é a camada de profundidade.

🇦🇷 Argentina 3–3 (4–2) França — Final

Evidence: redemption_journey, years_since_last_title, final_match, defending_champion_down

Com PlayerFacts: +Messi (legacy_complete: 98), +Messi (last_dance: 85), +Mbappé (hat_trick: 90), +Messi (captain_title: 80)

Nova evidence: "Messi conquista o único título que faltava. Mbappé faz hat-trick na final, o primeiro desde 1966."
Headline muda? ❌ Não. "Argentina campeã" continua sendo a história principal. O Why It Matters fica ENRIQUECIDO, não substituído.

🇧🇷 Brasil 1–1 (2–4) Croácia — Quartas

Hero atual: "Brasil, pentacampeão, é eliminado nos pênaltis pela Croácia" (confidence: 0.91)
Evidence: traditional_power_eliminated, isPopular, penalty_drama

Nota: Se a lesão do Neymar tivesse ocorrido ANTES da Copa, o PlayerFact "injury" teria disparado com weight altíssimo — possivelmente virando hero do dia.

● 8

Relatório Final

1. Vale a pena construir Player Narratives agora?
Sim, mas com escopo mínimo. O maior ponto cego do sistema atual (Princípio 6: Pessoa > Time) é resolvido com ~100 linhas de dataset + ~40 linhas de código. O retorno editorial é desproporcional ao investimento.
2. Quanto valor editorial isso adiciona?
+15-20% na qualidade do Why It Matters. Em 3 dos 3 casos testados, o hero não muda — mas o Why It Matters ganha profundidade. O ganho real está em dias onde o time story é fraco e o player story é forte. Exemplo: "Neymar lesionado" sem um jogo grande no mesmo dia viraria hero — hoje o sistema não teria hero nenhum.
3. Qual o MVP mínimo?
5 jogadores + 4 tipos de PlayerFact (41 linhas). Um dataset de 5-10 jogadores lendários (Messi, CR7, Neymar, Mbappé, Modric), com 4 detectores: last_dance, legacy_complete, hat_trick, injury. Integração direta no HistoricalContextBuilder como mais uma fonte de fatos. Zero arquitetura nova.
4. Qual a menor arquitetura possível?
lib/
  player-roster.ts     → 5-10 jogadores lendários + dados básicos
  player-facts.ts      → detecta last_dance, legacy_complete, hat_trick, injury
  historical-context-builder.ts → +1 linha: import playerFacts; adiciona ao array

A mesma arquitetura de historical-context-builder.ts. PlayerFacts entram como HistoricalFact[] no mesmo array. Zero mudança no EditorialStoryEngine. Zero mudança nos componentes.

5. O que NÃO devemos construir ainda?
  • Não criar banco de jogadores completo (não precisamos de todos os 736 jogadores da Copa)
  • Não criar NLP para declarações (exige redação, não algoritmo)
  • Não criar monitoramento de lesões em tempo real (API-Football já fornece)
  • Não criar tracking de recordes históricos completo (apenas os top 5 mais relevantes)
  • Não criar golden_boot tracking em tempo real (pode ser diário, não em tempo real)
